# ADR-011: Vector Search Strategy

## Status
Accepted

## Context
The platform requires semantic search across transcripts, stories, knowledge objects, and articles. Traditional full-text search (pg_trgm, Elasticsearch) cannot capture semantic meaning. We need vector embeddings for similarity search, clustering, and recommendation.

## Decision
We use Weaviate as a read-only vector database projection, populated via event-driven sync from PostgreSQL. Embeddings are generated by the embedding-service (Python).

## Rationale
- **Semantic search**: Vector embeddings capture meaning beyond keyword matching
- **Hybrid search**: Weaviate supports both vector and keyword (BM25) search
- **Multi-modal future**: Weaviate supports image and text vectors
- **Managed projection**: Like Neo4j, Weaviate is rebuilt from PostgreSQL

## Consequences
### Positive
- Semantic search across all content types
- Story similarity recommendations (Story DNA)
- Clustering for theme/motif discovery
- Rebuildable from PostgreSQL

### Negative
- Embedding generation cost (API calls to embedding models)
- Storage overhead for vector indexes
- Eventual consistency delay

## Embedding Strategy
| Content Type | Model | Dimensions | Chunking |
|-------------|-------|-----------|----------|
| Transcript segments | text-embedding-3-small | 1536 | ~512 tokens |
| Stories | text-embedding-3-small | 1536 | Full story |
| Knowledge objects | text-embedding-3-small | 1536 | Full object |
| Themes/Motifs | text-embedding-3-small | 1536 | Description |

## Sync Flow
```
PostgreSQL → Domain Event → embedding-service → Weaviate
```
